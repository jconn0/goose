# Milestone v4.0: Security, Performance & Coach Expansion

**Status:** ✅ SHIPPED 2026-06-06
**Phases:** 16–19
**Total Plans:** 12

## Overview

Harden the URL scheme security by blocking state-changing deep links, eliminate SwiftUI ObservableObject re-render overhead via full `@Observable` migration, expand the Coach tab to support four AI providers (ChatGPT, Claude, Custom endpoint, Gemini), and complete pt-PT localisation for all v4.0 UI additions.

### Phase 17: @Observable Migration

**Goal**: GooseAppModel and HealthDataStore use Swift @Observable macro — only views that access a changed property re-render
**Depends on**: Phase 16
**Requirements**: PERF-01, PERF-02, PERF-03
**Plans**: 4 plans (4 waves)

Plans:

- [x] 17-01-PLAN.md — Wave 1: GooseAppModel @Observable migration + @Environment rewire + MoreDataStore Combine removal
- [x] 17-02-PLAN.md — Wave 2: HealthDataStore @Observable migration + @State ownership + @ObservedObject removal
- [x] 17-03-PLAN.md — Wave 3: GooseBLEClient @Observable migration (NSObject kept) + MoreView onChange route status
- [x] 17-04-PLAN.md — Wave 4: GooseSwiftApp injection sweep + full build verification + PERF-03 runtime check

**Details:**

- `@Observable class GooseAppModel` — 68 `@Published` removed, all views using `@Environment(GooseAppModel.self)` pattern
- `@Observable class HealthDataStore` — `@ObservedObject` → `@State`; `@Bindable` introduced for Picker bindings; `nonisolated(unsafe)` on NSObjectProtocol observer for deinit cleanup
- `@Observable class GooseBLEClient` — NSObject conformance preserved (CBCentralManagerDelegate); `@Bindable` local var pattern for binding in View bodies
- `GooseSwiftApp`: `@State private var model = GooseAppModel()` + `.environment(model)` — no legacy wrappers
- PERF-03: NavigationRequestObserver warning eliminated by removing ObservableObject broadcast chain

### Phase 18: Coach Multi-Provider

**Goal**: Coach tab supports multiple AI providers and user-configured custom endpoints
**Depends on**: Phase 16
**Requirements**: COACH-01, COACH-02, COACH-03, COACH-04, COACH-05, COACH-06
**Plans**: 6 plans (6 waves)

Plans:

- [x] 18-01-PLAN.md — Wave 1: CoachProvider protocol + CoachProviderRegistry + CoachChatModel refactor + ChatGPTCoachProvider + Wave 0 test stubs (COACH-01, COACH-06)
- [x] 18-02-PLAN.md — Wave 2: ClaudeCoachProvider (Anthropic Messages API SSE + Keychain) (COACH-02, COACH-03)
- [x] 18-03-PLAN.md — Wave 3: CustomEndpointCoachProvider (OpenAI Chat Completions SSE + URL validation + Keychain) (COACH-02, COACH-04)
- [x] 18-04-PLAN.md — Wave 4: GeminiCoachProvider (Google OAuth PKCE via WKWebView + streamGenerateContent SSE) (COACH-02, COACH-03)
- [x] 18-05-PLAN.md — Wave 5: CoachSettingsSheet provider picker UI + gear icon + four-provider registry (COACH-05)
- [x] 18-06-PLAN.md — Wave 6: Integration, build/test verification, migration smoke test (COACH-01, COACH-05, COACH-06)

**Details:**

- `CoachProvider` protocol: `id`, `displayName`, `isAuthenticated`, `availablePresets`, `send(messages:systemPrompt:) -> AsyncStream<String>`, `signOut()`
- `CoachProviderRegistry`: manages all four providers, persists `activeProviderId` in UserDefaults
- `ChatGPTCoachProvider`: wraps existing `CodexSelfContainedAuthClient` with OAuth; existing token at `chatgpt-auth` works without re-auth
- `ClaudeCoachProvider`: Anthropic Messages API SSE streaming; API key in Keychain service `com.goose.swift.claude`
- `CustomEndpointCoachProvider`: OpenAI-compatible Chat Completions SSE; URL validation (https required for non-local); base URL + model in UserDefaults, API key in Keychain
- `GeminiCoachProvider`: Google OAuth 2.0 PKCE via WKWebView; streamGenerateContent SSE; client_id supplied by user
- `CoachSettingsSheet`: `Section("Provider")` → `ForEach(registry.allProviders)` → `CoachProviderPickerRow` → `registry.selectProvider(id:)`

Human verification deferred (device required):

2. Claude streaming with live Anthropic API key
3. Custom endpoint streaming with live HTTPS endpoint
4. Gemini OAuth + streaming with Google Cloud credentials

## Milestone Summary

**Key Accomplishments:**

1. Deep link security locked — external callers cannot trigger BLE writes or sensor captures via `gooseswift://` URL scheme
2. Full `@Observable` migration across GooseAppModel + HealthDataStore + GooseBLEClient — 68 `@Published` removed, per-property re-render granularity achieved
3. Four AI providers in Coach: ChatGPT (existing), Claude (Anthropic API), Custom endpoint (OpenAI-compatible), Gemini (Google OAuth PKCE)
4. Coach settings UI with provider picker, per-provider config sheets, Keychain-backed API key storage
5. Complete pt-PT localisation for all v4.0 additions (128 strings total, including 9 fixed post-audit)
6. App startup non-blocking: overnight recovery dispatched to background queue; onboarding skip button available

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| @Observable migration before Coach expansion | Phase 17 before Phase 18 — CoachView must use `@Environment(GooseAppModel.self)` pattern | ✓ Good — CoachView correctly consumes @Observable pattern |
| Four providers in Wave approach | Each provider independent; added sequentially to avoid merge conflicts on CoachProviderRegistry | ✓ Good — clean wave execution |
| Google OAuth via WKWebView (no SDK) | No Google SDK dependency; user-supplied client_id; PKCE mandatory (no implicit flow) | ✓ Good — zero SDK dependency |
| L10N-03 gap closure inline (no new phase) | 9 missing strings fixed via direct xcstrings edit post-audit (commit 0f2ea15) | ✓ Good — faster than a new phase |

**Issues Resolved:**

- NavigationRequestObserver multiple-update-per-frame warning eliminated by removing ObservableObject broadcast chain
- Phase 19 onboarding Keychain state leak: `onboardingComplete` no longer restored on reinstall
- UX-01: skip button string architecture corrected (hardcoded Portuguese → proper xcstrings key)
