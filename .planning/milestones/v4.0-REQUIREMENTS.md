# Requirements Archive: Goose v4.0 — Security, Performance & Coach Expansion

**Archived:** 2026-06-06
**Status:** ✅ SHIPPED — all requirements implemented (COACH-02/06 partial — device verification deferred)

## Milestone Goal

Harden the URL scheme security, eliminate SwiftUI re-render overhead via @Observable migration, and expand the Coach tab to support multiple AI providers and user-configured endpoints.

## Requirements

### Security

- [x] **SEC-01**: Deep links via `gooseswift://` URL scheme can only invoke **read-only** debug commands; state-changing commands (Bluetooth writes) are blocked from external callers (upstream PR #15 by kobemartin)
  - **Outcome:** SATISFIED — `allowsRemoteInvocation` property on GooseBLETypes; guard at GooseAppModel+Lifecycle.swift line 73. Survives Phase 17 and 18 edits intact.

### Performance

- [x] **PERF-01**: `GooseAppModel` uses Swift `@Observable` macro — views that do not access a changed property do not re-render (eliminates `objectWillChange` broadcast)
  - **Outcome:** SATISFIED — `@MainActor @Observable final class GooseAppModel`; `GooseSwiftApp` uses `@State` + `.environment(model)`; all views use `@Environment(GooseAppModel.self)`

- [x] **PERF-02**: `HealthDataStore` uses Swift `@Observable` macro — same per-property tracking benefit
  - **Outcome:** SATISFIED — `@MainActor @Observable final class HealthDataStore`; `AppShellView` uses `@State`

- [x] **PERF-03**: `Update NavigationRequestObserver tried to update multiple times per frame` warning eliminated at capture startup
  - **Outcome:** SATISFIED (structural) — No `NavigationRequestObserver` references in codebase; ObservableObject broadcast chain removed. Runtime confirmation pending (manual developer step).

### Coach Expansion

- [x] **COACH-01**: `CoachProvider` protocol abstracts the AI provider — `send(messages:systemPrompt:) async throws -> AsyncStream<String>`
  - **Outcome:** SATISFIED — `CoachProvider` protocol in `CoachProviderProtocol.swift`; all four providers conform; `CoachChatModel` routes through `registry.activeProvider`

- [x] **COACH-02**: Multiple named accounts per provider stored in Keychain with provider prefix
  - **Outcome:** PARTIAL — Each provider uses isolated Keychain entries (adequate isolation) but no formal named-account abstraction. No collision risk identified. Device verification deferred.

- [x] **COACH-03**: At least one additional provider supported (Claude API by Anthropic)
  - **Outcome:** SATISFIED — `ClaudeCoachProvider` streams from Anthropic Messages API; Keychain via `ClaudeKeychain`. Three additional providers total: Claude, Custom endpoint, Gemini.

- [x] **COACH-04**: User-configured custom endpoint (OpenAI Chat Completions-compatible `POST /v1/chat/completions` with SSE streaming)
  - **Outcome:** SATISFIED — `CustomEndpointCoachProvider`; HTTPS validation; base URL + model in UserDefaults, API key in Keychain

- [x] **COACH-05**: Provider picker UI in More/Coach settings — shows configured accounts, lets user add/remove/select active account
  - **Outcome:** SATISFIED — `CoachSettingsSheet` with `Section("Provider")` → `ForEach(registry.allProviders)` → `CoachProviderPickerRow` → `registry.selectProvider(id:)`

- [x] **COACH-06**: Existing single OpenAI key migrated to named account on first launch after upgrade
  - **Outcome:** PARTIAL — `ChatGPTCoachProvider` uses existing `CodexEmbeddedAuth` with `account = "chatgpt-auth"` (unchanged from pre-Phase 18). No explicit migration code; existing token assumed to work transparently. Cold-launch verification deferred to device.
